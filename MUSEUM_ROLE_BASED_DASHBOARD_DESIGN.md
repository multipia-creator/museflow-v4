### 1.2 역할별 상세 정의

#### 1️⃣ 관장 (Director)
```yaml
역할 코드: director
권한 레벨: 10 (최고)
주요 책임:
  - 뮤지엄 전체 운영 총괄
  - 전략적 의사결정
  - 예산 승인 및 관리
  - 주요 전시 기획 승인
  - 인사 관리
  
필요한 정보:
  - 전체 부서 실적 요약
  - 예산 집행 현황
  - 방문자 통계 (연간/월간/주간)
  - 주요 KPI 대시보드
  - 진행 중인 프로젝트 현황
  - 긴급 이슈 및 알림
  
접근 권한:
  - 모든 데이터 조회 가능
  - 승인 권한 (예산, 전시, 인사)
  - 보고서 열람
  - 감사 로그 조회
```

#### 2️⃣ 학예실장 (Chief Curator)
```yaml
역할 코드: chief_curator

필요한 정보:

  - 소장품 현황 및 상태
  - 학예사별 업무 진행도
  - 보존처리 현황
  - 학술 연구 진행 상황

접근 권한:
  - 학예 부서 모든 데이터

## 🔐 4. 권한 레벨 및 데이터 접근 제어

### 4.1 권한 레벨 정의
```
Level 10: 관장 (Director)
  - 모든 데이터 조회/수정/삭제 가능
  - 최종 승인 권한
  - 감사 로그 조회
  - 시스템 설정 변경

Level 8: 학예실장 (Chief Curator)
  - 학예 부서 전체 데이터
  - 부서 내 승인 권한
  - 학예사 관리 권한
  - 전시 승인 권한

Level 6: 학예사, 레지스트라 (Curator, Registrar)
  - 담당 업무 데이터 조회/수정
  - AI Agent 사용 권한
  - 협업 기능 사용
  - 제한적 승인 권한

Level 5: 에듀케이터, 행정지원, 시설관리
  - 담당 부서 데이터
  - 부서 내 협업
  - 보고서 작성
  - 제한적 조회 권한

Level 4: 전시해설사, 강사 (Docent, Instructor)
  - 자신의 업무 데이터만
  - 필요한 자료 조회
  - 피드백 입력
  - 제한적 기능
```

### 4.2 데이터 접근 매트릭스
```
┌─────────────┬────┬────┬────┬────┬────┬────┬────┬────┬────┐
│ 데이터/역할 │관장│학장│학사│레지│에듀│해설│강사│행정│시설│
├─────────────┼────┼────┼────┼────┼────┼────┼────┼────┼────┤
│ 전체 예산    │ RW │ R  │ R  │ R  │ R  │ -  │ -  │ RW │ -  │
│ 부서 예산    │ RW │ RW │ R  │ R  │ RW │ -  │ -  │ RW │ RW │
│ 전시 데이터  │ RW │ RW │ RW │ R  │ R  │ R  │ -  │ R  │ -  │
│ 소장품 DB    │ R  │ RW │ RW │ RW │ R  │ R  │ -  │ -  │ -  │
│ 교육 프로그램│ R  │ R  │ -  │ -  │ RW │ R  │ RW │ R  │ -  │
│ 인사 데이터  │ RW │ R  │ R  │ R  │ R  │ R  │ R  │ RW │ R  │
│ IoT 센서    │ R  │ R  │ -  │ -  │ -  │ -  │ -  │ -  │ RW │
│ 감사 로그    │ R  │ -  │ -  │ -  │ -  │ -  │ -  │ R  │ -  │
└─────────────┴────┴────┴────┴────┴────┴────┴────┴────┴────┘

R = Read (조회)
W = Write (수정)
RW = Read + Write

## 🔄 6. 데이터베이스 스키마 변경

### 6.1 users 테이블 업데이트
```sql
-- 기존 role 필드 타입 변경
ALTER TABLE users ADD COLUMN new_role TEXT;
UPDATE users SET new_role = role;
ALTER TABLE users DROP COLUMN role;
ALTER TABLE users RENAME COLUMN new_role TO role;

-- 권한 레벨 추가
ALTER TABLE users ADD COLUMN permission_level INTEGER DEFAULT 4;

-- 부서 정보 추가
ALTER TABLE users ADD COLUMN department TEXT;

-- 역할별 권한 레벨 설정
UPDATE users SET permission_level = 10 WHERE role = 'director';
UPDATE users SET permission_level = 8 WHERE role = 'chief_curator';
UPDATE users SET permission_level = 6 WHERE role IN ('curator', 'registrar');
UPDATE users SET permission_level = 5 WHERE role IN ('educator', 'admin_support', 'facility_manager');
UPDATE users SET permission_level = 4 WHERE role IN ('docent', 'instructor');
```

### 6.2 새로운 테이블
```sql
-- 승인 워크플로우
CREATE TABLE IF NOT EXISTS approvals (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  type TEXT NOT NULL, -- 'budget', 'exhibition', 'loan'
  requester_id INTEGER,
  approver_id INTEGER,
  status TEXT DEFAULT 'pending', -- 'pending', 'approved', 'rejected'
  amount REAL,
  description TEXT,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  approved_at DATETIME,
  FOREIGN KEY (requester_id) REFERENCES users(id),
  FOREIGN KEY (approver_id) REFERENCES users(id)
);

-- 알림
CREATE TABLE IF NOT EXISTS notifications (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  user_id INTEGER,
  type TEXT NOT NULL, -- 'info', 'warning', 'urgent'
  title TEXT,
  message TEXT,
  read BOOLEAN DEFAULT 0,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (user_id) REFERENCES users(id)
);

-- 부서
CREATE TABLE IF NOT EXISTS departments (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  name TEXT NOT NULL,
  head_id INTEGER, -- 부서장
  budget REAL,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (head_id) REFERENCES users(id)
);
```
